# PagesでHEDGELOOP機能を初期テンプレとして網羅・上位互換化する設計書 v2

## 0. 今回の改訂方針

この版では、前版から以下を明確に修正する。

- **Premiere拡張は対象外** とする
- **案件管理は /contents と重複させない**
- Pagesは「業務実行画面」ではなく、**運用パック / テンプレ / ルール / 台帳 / ガイド / スニペットの層** に徹する
- HEDGELOOPの良い機能は吸収するが、既存の基幹機能を侵食しない

この設計の主張は単純で、**案件の実データは /contents に集約、Pagesはその運用を強くするテンプレ基盤にする** ということ。

---

## 1. 目的

既存のSNS Ops SaaSの基幹機能はそのまま磨きつつ、HEDGELOOPが持つ動画編集ディレクション向けの価値を、Pagesの初期公式テンプレ群で吸収する。

狙いは以下。

- HEDGELOOPの主要価値をPagesテンプレでカバーする
- ただし案件の登録・更新・状態遷移の実体は /contents に一本化する
- ユーザーがテンプレを1クリック追加するだけで「新機能が増えた」と感じる体験を作る
- 将来のテンプレ共有・配布・販売まで繋がる基盤にする

## 2. プロダクト原則

### 2.1 変えないもの
既存の中核機能はプロダクトの本丸として維持する。

- `/home`：経営モードのダッシュボード
- `/contents`：1行=1コンテンツの制作進行シート
- `/billing` / `/invoices`：請求生成・請求書管理
- `/vendors` / `/payouts`：外注・支払い管理
- `/notifications`：通知センター
- `/settings`：組織・権限・移行・監査

### 2.2 Pagesの役割
PagesはNotion風ドキュメントではあるが、単なるメモ置き場ではない。役割は **機能拡張OSレイヤー**。

ただし、ここでいう「機能拡張」は**実データの二重管理を増やすことではない**。

Pagesが担うのは以下。

- 運用ルール
- 手順書
- 台帳
- チェックリスト
- 修正指示テンプレ
- チーム別ルール
- クライアント別運用パック
- チャット通知文面
- オンボーディング / 権限ガイド
- 実画面への導線ハブ

### 2.3 /contentsとの責務分離
ここが最重要。

#### /contentsが持つもの
- 1行=1コンテンツの実体データ
- 納期
- ステータス
- 担当進行に必要な主要列
- 請求 / 外注 / 支払いへの連動元データ

#### Pagesが持つもの
- どう運用するか
- どう入力するか
- 誰が何を確認するか
- どういう文言で指示するか
- チーム内での意味づけ
- 運用時の判断基準
- 関連画面への導線

つまり、**Pagesは業務のOS、/contentsは業務の台帳** である。

## 3. HEDGELOOP機能の吸収方針（重複回避版）

HEDGELOOPで見えている主要価値を、Pagesでどう吸収するかを整理する。

### 3.1 メンバー管理
HEDGELOOP側の価値:
- 編集者 / ディレクター登録
- 役割別の並び替え
- チーム共有

吸収方針:
- 人事マスタの厳密管理ではなく、**運用台帳テンプレ** としてPagesに載せる
- 実際の権限や組織メンバーは `/settings/members` 側が正式なソース
- Pages側は「アサイン判断に使う運用情報」を持つ

Pagesテンプレで持つ内容:
- 名前
- 呼称
- 役割
- 得意ジャンル
- 使用ソフト
- 稼働可能曜日
- 稼働量目安
- 単価帯メモ
- 注意点
- 相性メモ
- 依頼時テンプレ

### 3.2 案件管理
HEDGELOOP側の価値:
- 案件テンプレ登録
- 新規案件作成時に案件名・担当・素材リンク等を登録
- スプレッドシートからコピペ移行

吸収方針:
- **案件の実登録は /contents のみ**
- Pagesでは案件情報そのものを持たない
- Pagesでは、案件をどう作るか・どう命名するか・どう入力するかの**テンプレとルールのみ**を持つ

Pagesテンプレで持つ内容:
- 案件命名規則
- コンテンツタイトル命名例
- 編集者 / 確認者の割当ルール
- 素材リンク整理ルール
- クライアント別の入力注意点
- `/contents` 入力手順
- `/contents` に貼るためのコピペ雛形
- 初回案件登録チェックリスト

### 3.3 ステータス管理
HEDGELOOP側の価値:
- ステータスの追加 / 並び替え / 削除
- チームごとの独自運用

吸収方針:
- DB上のstatus enumや基幹ステータス設計は /contents 側で統一する
- Pagesでは、**そのチームにおける意味づけ・運用ルール** を管理する
- ステータスをPages上で勝手に増やすのではなく、既存statusと運用意味を結ぶ

Pagesテンプレで持つ内容:
- ステータス名
- その意味
- 次に動く人
- 完了条件
- 例外処理
- よくある詰まりポイント
- 標準statusとの対応表

### 3.4 修正指示・チェック
HEDGELOOP側の価値:
- 秒数つき修正コメント
- テンプレ文面
- 編集者に共有し、チェック進捗を見る

吸収方針:
- Pagesで最も強く吸収する領域
- 修正指示テンプレ、チェック観点、共有文面、運用ルールを公式テンプレ化する
- ただしリアルタイム進捗管理やタスクチェックの実体をPagesに持たせすぎない
- Pagesは「指示の質と速度を上げる層」とする

Pagesテンプレで持つ内容:
- 秒数指定の書き方
- 修正指示テンプレ集
- よくあるNG例 / 良い例
- ディレクター確認観点
- クライアント向け言い換え例
- 編集者共有文面
- 修正完了確認チェックリスト

### 3.5 ChatWork連携
HEDGELOOP側の価値:
- APIトークン設定
- デフォルト送信ルーム
- プロジェクト別送信先
- 自動送信ON/OFF

吸収方針:
- 連携の実機能は本体側で持つ余地を残しつつ、Pagesでは**運用セットアップと文面標準化**を提供する
- Pagesは通知ルール・ルーム設計・文面テンプレのハブになる

Pagesテンプレで持つ内容:
- API取得手順
- ルーム命名規則
- デフォルト送信先の決め方
- 案件別通知の設計ルール
- 毎朝リマインド文面
- 納期遅延催促文面
- 提出前確認文面

### 3.6 チーム設定
HEDGELOOP側の価値:
- チーム作成
- チームコード参加
- チーム共有
- チーム離脱

吸収方針:
- チーム作成や権限は `/settings` が正式導線
- Pagesでは **オンボーディングパック** として、参加後に迷わないようにする

Pagesテンプレで持つ内容:
- 新メンバー向け案内
- 役割説明
- 最初に見るページ
- よくある質問
- 運用ルールリンク集
- 連絡先とエスカレーション先

### 3.7 対象外
- Premiere拡張
- Premiere拡張インストール手順
- Premiere用マーカー連携

この領域は今回の初期テンプレ設計から外す。

## 4. Pagesで実現するべき体験

### 4.1 ユーザー体感
ユーザーは「ページを作る」のではなく、**運用パックを導入する**。

理想体験:
1. Pagesの `＋テンプレを導入` を押す
2. 公式テンプレ一覧を見る
3. 目的に合うテンプレを選ぶ
4. 1クリックで導入する
5. Pages一覧に複数ページがまとまって生成される
6. 見た瞬間に「編集ディレクション機能が増えた」と感じる

### 4.2 Pagesがやってはいけないこと
- /contents と同じ案件一覧を持つ
- ステータス実体を二重管理する
- メンバーマスタを二重管理する
- 請求や支払いの実データを持つ
- 競合の画面をそのまま複製する

### 4.3 Pagesがやるべきこと
- 実務の型を配る
- 入力ミスを減らす
- 指示の質を上げる
- チームごとの差を標準化する
- 本体画面への導線を短くする

- 並び順の意味づけ

本体側に残すもの:
- 組織メンバー実体
- 権限
- 参加 / 承認 / 離脱

Pagesでカバーするもの:
- 登録ルール
- 入力テンプレ
- 命名規則
- 必須リンク
- 担当割当ルール

本体側に残すもの:
- 案件実データ
- コンテンツ行の追加

### 6.3 ステータス管理
Pagesでカバーするもの:
- チームにおける意味づけ
- ステータス遷移ルール
- 完了条件

本体側に残すもの:
- statusの保存
- status選択UI
- レポート / 集計

Pagesでカバーするもの:
- 指示テンプレ
- チェック観点

- チーム標準ルール

本体側に残すもの:
- 実際の進捗同期や専用レビュー画面が必要なら将来別実装

### 6.5 ChatWork
Pagesでカバーするもの:
- 設定ガイド
- 文面テンプレ
- 送信ルール

本体側に残すもの:
- 連携設定そのもの

## 7. テンプレ導入UX設計

### 7.1 追加導線
Pages欄のCTAを以下にする。

- 主CTA：`＋テンプレを導入`
- 副CTA：`空ページを作成`

### 7.2 テンプレ選択モーダル
表示項目:
- テンプレ名
- カテゴリ
- 公式ラベル
- 何が改善されるか
- 含まれるページ数
- 本体との接続先
- プレビュー
- 導入ボタン

### 7.3 ラベル設計
- 公式
- 上級
- 編集ディレクション
- メンバー運用
- 修正管理
- 通知運用
- 1クリック導入

### 7.4 導入時オプション
- そのまま追加
- 名前を変えて追加
- サンプル記入あり / 空欄版を選択
- ルートページ配下にまとめる

### 7.5 導入後の見せ方
例:
- 編集ディレクションOS
  - 01_全体フロー
  - 02_役割分担
  - 03_ステータス運用ガイド
  - 04_修正指示テンプレ
  - 05_初稿チェック観点

これにより「複数機能が増えた感」を出す。

## 11. 非機能要件

### 11.1 速度
- テンプレ追加は体感1秒台目標
- 複数ページ生成でも先に成功体験を返す

### 11.2 監査性
- どのテンプレがいつ追加されたか残す
- 公式テンプレからの改変有無を追跡可能にする

### 11.3 安全性
- 同名衝突回避
- アーカイブ復元
- 誤導入後の削除容易化

### 11.4 保守性
- テンプレ定義はJSON / DBで管理

## 15. この設計で勝てる理由

- 競合の良いところを、基幹機能を壊さずに吸収できる
- 案件実体を /contents に集約するので、請求・支払いとの一気通貫を失わない
- Pagesで運用の型を配れるので、現場導入が速い
- メンバー運用、修正品質、通知運用、クライアント知見がテンプレとして蓄積する
- 競合が画面で増やしている価値を、こちらはテンプレ資産として配布できる
