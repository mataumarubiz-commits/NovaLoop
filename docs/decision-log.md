# Decision Log
D342 Projects should default to one low-cognitive-load operational surface, not a control-center dashboard (2026-04-08): Operator feedback made it clear that the prior `/projects` screen surfaced too many concerns at once: KPI summary, creation flows, rescue flows, saved views, filters, finance, and editing all competed on the first paint. Policy in this round: `/projects` now defaults to a single operational surface with a sticky header, one searchable table as the main workspace, an optional calendar toggle, and a right-side detail drawer for deeper work. The default summary is limited to six operational counts (`今日納期`, `今週納期`, `遅延案件`, `修正対応中`, `素材待ち`, `外注提出待ち`), while advanced filters, column customization, saved views, and orphan-row rescue remain available only behind a lower-emphasis details panel. Finance is intentionally kept out of the default surface and appears only in optional columns or inside the detail drawer.
D343 Projects visibility should be admin-only, matching the owner/executive_assistant operational scope (2026-04-08): The redesigned Projects workspace now centralizes billing-affecting project operations and high-level operational triage, so exposing it to general `member` users would both add noise and widen access beyond the requested scope. Policy in this round: `/projects` and `/projects/[id]` are treated like an admin/assistant operational surface and are available only to `owner` and `executive_assistant`; sidebar navigation, quick-create actions, and Home shortcuts must therefore suppress `/projects` for `member` users rather than showing a dead-end route.
D344 Projects should show simplified display status plus derived helper tags rather than exposing the raw workflow taxonomy (2026-04-08): The underlying project/content records still keep the richer workflow states needed for automation and compatibility, but showing those raw states directly on the main Projects screen increases cognitive load and encourages users to manage the system through too many labels. Policy in this round: the `/projects` board presents only six primary display statuses (`未着手`, `進行中`, `提出待ち`, `修正中`, `完了`, `保留`) and derives helper tags such as `素材待ち`, `先方待ち`, `外注待ち`, `仕様変更`, and `納期危険` from existing project/content/change signals. We intentionally do not introduce a new writable project-tag schema in this round; helper tags remain derived so the operator sees a simpler surface without creating a second state model to maintain.
