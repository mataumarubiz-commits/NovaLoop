# 月次締め自動化 設計書 v2

##目的
受託会社における月末の請求・外注支払い・経費・証憑確認・粗利確認を、案件進行データから自動生成し、経理専任を置かずに owner または assistant が例外確認だけで締められる状態を作る。

##前提
- 既存プロダクトの核は「/contents を中心にした制作進行」と「請求/外注支払い/ページ/通知/移行」である
- 本設計は既存の業務OS設計を破壊せず、その上に「締めエンジン」を足す
- 汎用会計SaaSを作るのではなく、受託業界特化の月次締めOSを作る
- 会計ソフトとの関係は「このSaaSが業務の正本、freee は最終出力先」とする

##この設計で狙う到達点
- 月次請求対象の抽出を人手でやらない
- 外注支払い対象と金額の算出を人手でやらない
- 案件直結経費の回収と証憑保管をここで完結させる
- 締めを止めている未処理を1画面で見えるようにする
- 案件別粗利を自動算出し、経営判断までつなげる

##ヒアリング反映済みの前提
- 最初に潰す痛みは「領収書回収」である
- 経費入力は MVP では owner / executive_assistant のみで行う
- 請求対象ステータスは delivered / published / rejected を維持する
- 外注支払い対象は請求対象 content のみとする
- 共通費は MVP では扱わない
- freee 連携は初期から API 連携前提で設計する
- 主訴求は /close に寄せる

##今回の結論
本当に実装すべきテーマは「経費管理」ではなく「月次締め自動化」である。

その中身は以下の5機能に収束する。

1. 請求自動判定エンジン
2. 外注支払い自動算出エンジン
3. 証憑付き経費回収基盤
4. Close Cockpit
5. 案件別粗利台帳

##スコープ
###MVPでやる
- /contents から請求候補を自動生成
- content 単位で外注支払予定額を自動生成
- 案件直結経費のみを登録できる
- 領収書アップロードと OCR/AI 読取
- 経費の client / project / content 紐付け
- 領収書未回収を前提にした催促・未処理可視化
- 月次の未処理一覧
- 案件別の売上 / 外注費 / 経費 / 粗利表示
- freee API 連携の最小実装

###MVPでやらない
- 複式簿記
- 汎用仕訳帳
- 銀行口座連携
- 消費税のフルロジック
- 減価償却
- 汎用経費精算ワークフロー
- 一般企業向け部門会計
- 共通費の配賦
- member 向け経費申請フロー

##ユーザーと役割
###owner
- 全ての締め処理の最終承認者
- 例外処理の判断者
- 締め完了の実行者

###executive_assistant
- 日常の登録、差分確認、証憑確認、支払い確認を行う
- 月次締めの実務担当

###member
- 当面は制作進行中心
- 将来は自分の立替経費申請や領収書提出のみ可能に拡張余地あり

##プロダクトの核となる思想
###1. 経理データは後入力しない
案件進行のイベントから請求・支払い・経費・証憑状態を生成する。

###2. 人がやるのは例外確認だけ
通常処理は自動、例外だけをキューに出す。

###3. 正しい入力単位は content
1行=1コンテンツを売上・外注費・経費の基準単位にする。

###4. 経費は会計のためではなく粗利のために扱う
案件別損益を確定するための経費を優先する。

##コアユースケース
###ユースケース1: 月次請求の自動生成
1. owner または assistant が target_month を指定
2. ルールに合致する contents を抽出
3. client 別に invoice draft を自動作成
4. 請求漏れ候補と例外を表示
5. PDF を生成して保存
6. freee 用 CSV を出力

###ユースケース2: 外注支払い予定の自動生成
1. content に紐づく vendor / pricing rule を参照
2. 対象月の支払い候補を自動算出
3. vendor 別に集計
4. vendor 請求の登録内容と差分照合
5. 承認後に payout を生成

###ユースケース3: 経費と証憑の回収
1. assistant が現場担当から領収書を回収する
2. 領収書画像または PDF をアップロードする
3. OCR/AI が日付、金額、支払先、カテゴリ候補を抽出する
4. client / project / content の紐付け候補を出す
5. 不明点だけを人が修正する
6. 未提出領収書を /close で催促対象として可視化する
7. 締め対象月の経費に反映する

###ユースケース4: 月次締め
1. /close で target_month を開く
2. 未請求、外注差分、未証憑、案件未紐付けなどを確認
3. 例外だけ処理
4. 締め完了
5. CSV エクスポート

###ユースケース5: 案件別粗利確認
1. client または project を選択
2. 売上、外注費、経費、粗利額、粗利率を確認
3. 赤字案件や粗利悪化案件を検知
4. 値上げ、運用設計変更、外注単価見直しの判断に使う

##機能要件
###1. 請求自動判定エンジン
####目的
請求対象の抽出を自動化し、請求漏れをなくす。

####入力
- target_month
- delivery_month
- status
- billable_flag
- client_id
- project_name
- unit_price
- client別請求ルール

####ルール
- デフォルトは delivery_month = target_month
- status は delivered / published / rejected のいずれかで対象
- billable_flag = true のみ対象
- client ごとに対象 status を上書き可
- client ごとに締め日、支払期限、請求書タイトル規則を設定可

####出力
- invoice draft
- invoice lines
- 請求漏れ候補
- 差分一覧

####例外
- 金額0
- client未設定
- billable_flag=false だが請求候補に見えるもの
- 同一 content の二重請求疑い

###2. 外注支払い自動算出エンジン
####目的
外注請求待ちの集計業務をなくし、先に支払予定額を自動で作る。

####入力
- content_id
- vendor_id
- role
- pricing_type
- unit_price
- quantity
- option_flags
- override_amount
- billing linkage

####pricing_type 候補
- per_content
- per_edit
- per_thumbnail
- per_shoot
- fixed_monthly
- override

####出力
- vendor別支払予定額
- vendor別明細
- vendor invoiceとの差分
- payout 候補

####例外
- vendor 未設定
- 単価ルール未設定
- vendor請求との金額不一致
- 二重支払い疑い

###3. 証憑付き経費回収基盤
####目的
経費を単なる台帳ではなく、締め対象の原価データとして扱う。
また最重要課題である「領収書回収漏れ」を減らすため、回収漏れの検知と催促導線を最優先で実装する。

####対象経費
- 撮影交通費
- ロケ費
- 素材購入費
- 代理購入費
- 案件直結ツール費

####入力方法
- 手入力
- 画像/PDFアップロード
- CSV取込
- 将来: メール転送

####回収フロー
- 締め対象月の content をもとに「領収書が発生しそうな案件」を抽出できるようにする
- assistant が未提出の案件を一覧で確認できる
- 任意メモで「誰に催促するか」「いつ催促したか」を残せる
- 将来は Slack 通知やフォーム提出へ拡張できるように payload を持たせる

####AI読取項目
- expense_date
- amount
- payee_name
- category candidate
- note candidate

####紐付け先
- client_id nullable
- project_name nullable
- content_id nullable
- org common expense

####未処理として扱う条件
- receipt なし
- amount 未確定
- category 未確定
- project/content 未紐付け
- target_month 未確定
- 回収依頼済みだが未提出

##自動化ルール
###請求ルール
- delivery_month が対象月
- billable_flag=true
- status が client rule に合致
- 同一 content が未請求

###外注ルール
- 請求対象 content に紐づく vendor assignment を対象
- override_amount があれば優先
- なければ pricing_type で算出
- vendor請求と差分があれば check 発火

###経費ルール
- receipt_path がないものは close blocked 候補
- receipt_collection_status = overdue のものは優先度 high の check を発火
- content_id があるものは案件直結経費として粗利へ直結
- client_id / project_name のみなら project原価に加算
- 共通費は MVP では扱わない

###締めルール
以下が0件なら close completed 可能
- open closing_checks
- draft expense
- 未承認 vendor 差分
- 未生成 invoice

###3. 請求対象ステータス
暫定値のまま delivered / published / rejected を採用する。

###4. 外注支払い基準
請求対象 content のみを対象とする。

###5. 共通費

###10. freee の正本

##これで確定したプロダクトの芯
このプロダクトの MVP は「経費入力ツール」ではない。

**月末に、請求・外注支払い・領収書回収・証憑確認・freee反映までを、owner または assistant が 1画面で例外処理だけして終わらせる Close OS である。**

##運用フローの確定版
###月中
- /contents で通常の制作進行を回す
- content と vendor assignment が蓄積される
- 案件直結経費が発生したものだけ手動または証憑アップロードで登録される

###月末
1. target_month を選ぶ
2. /close が請求候補、外注支払い候補、未領収書候補、未紐付け経費、freee未同期を自動集計する
3. assistant が社内担当と外注へ領収書回収の催促を行う
4. 回収後に OCR/AI で読取、案件紐付けを確認する
5. 請求 draft と payout draft を確認する
6. freee へ API 同期する
7. open check が 0 件なら締め完了にする

##月次締めロジック（確定）
###対象月の考え方
- target_month は `YYYY-MM`
- 請求・外注・経費・close_checks はすべて target_month を持つ
- 月末一括回収のため、経費は原則として締め対象月にまとめて処理する

###請求自動判定
対象条件:
- `contents.delivery_month = target_month`
- `billable_flag = true`
- `status in client_billing_rules.eligible_statuses`
- 同一 content が未請求

除外条件:
- unit_price is null or unit_price = 0
- client_id is null
- すでに invoice_lines.content_id に採番済み

出力:
- invoices
- invoice_lines
- closing_checks(check_type='invoice_candidate_missing' など)

###外注支払い自動算出
対象条件:
- 請求対象となった content のみ
- content_vendor_assignments が存在する

算出式:
- override_amount があればそれを優先
- `per_content` = unit_price * quantity
- `per_thumbnail` = option_flags.thumbnail = true のとき unit_price
- `per_shoot` = option_flags.shoot = true のとき unit_price
- `fixed_monthly` は vendor 単位で target_month に 1件のみ

差分:
- vendor_invoices が存在する場合は明細合計と自動算出額を比較
- 不一致時は closing_checks(check_type='vendor_amount_diff') を作成

###領収書回収ロジック
月末一括回収を前提に、未領収書候補を次の2系統で作る。

####社内担当向け候補
- expense はあるが receipt_path がない
- expense.status in ('draft','requested')
- receipt_collection_status in ('requested','overdue')

####外注向け候補
- vendor_invoice があるが pdf_path がない
- または支払い関連の証憑が不足している

####overdue 条件
- target_month の締め処理開始後も receipt_path がない
- receipt_requested_at から設定日数を超過

###締め判定
close completed 可能条件:
- open な closing_checks が 0
- expenses.status in ('draft','requested') が 0
- freee_sync_status = failed が 0
- invoice draft / payout draft の未確認が 0

close blocked 条件:
- 未領収書が残っている
- freee 同期 failed が残っている
- high severity の差分が残っている

**Given** target_month に請求条件を満たす content がある  
**When** invoices auto-generate を実行する  
**Then** clients ごとに invoices が作成される  
**And** invoice_lines に content が明細化される  
**And** 同一 content は二重請求されない

###TC-08 請求対象外 content は除外される
**Given** billable_flag = false または unit_price = 0 の content がある  
**When** invoices auto-generate を実行する  
**Then** invoices / invoice_lines に含まれない  
**And** warning または closing_check が作成される

###TC-09 payout 候補を自動生成できる
**Given** 請求対象 content に vendor assignment がある  
**When** payouts auto-generate を実行する  
**Then** vendor ごとに payout 候補が作成される  
**And** override_amount が優先される

###TC-10 vendor 請求との差分を検出できる
**Given** 自動算出額と vendor_invoice.total_amount が異なる  
**When** close checks を再計算する  
**Then** vendor_amount_diff の closing_check が生成される  
**And** /close の外注差分タブに表示される

###TC-11 freee に invoice を同期できる
**Given** freee 接続済みで partner mapping がある  
**When** freee sync invoices を実行する  
**Then** freee_sync_logs に synced が記録される  
**And** 対象 invoice の同期状態が更新される

###TC-12 freee に expense を同期できる
**Given** approved の expense がある  
**When** freee sync expenses を実行する  
**Then** freee_expense_id が保存される  
**And** freee_sync_status が synced になる

###TC-13 freee に payout を同期できる
**Given** approved の payout がある  
**When** freee sync payouts を実行する  
**Then** freee_sync_logs に結果が残る  
**And** failed 時は /close に表示される

###TC-14 freee 同期失敗を再実行できる
**Given** freee_sync_status = failed の entity がある  
**When** freee 再同期を実行する  
**Then** 再送される  
**And** 成功すれば failed から synced に変わる

###TC-15 締め完了をブロックできる
**Given** open の closing_check が1件以上ある  
**When** close complete を実行する  
**Then** status = blocked になる  
**And** 完了できない理由が表示される

###TC-16 条件を満たせば締め完了できる
**Given** open check = 0 かつ freee failed = 0 かつ未確認 draft = 0 である  
**When** close complete を実行する  
**Then** close_runs.status = completed になる  
**And** 監査ログが記録される

###TC-17 /profitability で案件別粗利が見える
**Given** invoices / payouts / expenses が揃っている  
**When** /profitability を開く  
**Then** client / project / content 粒度で粗利が見える  
**And** 粗利 = 売上 - 外注費 - 経費 で一致する

###TC-18 member は経理画面にアクセスできない
**Given** member ロールのユーザーがいる  
**When** /close /expenses /profitability へアクセスする  
**Then** 403 または権限エラーになる

##Claude Code / Codex への依頼テンプレ
###テンプレ1: migration 実装
- 目的: 指定 Epic の migration を追加する
- 前提: 既存 schema を壊さず差分追加
- 出力: SQL migration, rollback 方針, index, RLS policy 案

###テンプレ2: API 実装
- 目的: 指定 API Route を Next.js + Supabase 前提で実装する
- 前提: owner / assistant のみ許可
- 出力: handler, validation, error response, audit log, 型定義

###テンプレ3: UI 実装
- 目的: 指定画面を Addness 風の高品質 UI で実装する
- 前提: 情報密度は高いが視認性を崩さない
- 出力: page component, state, table, right pane, loading / empty / error state

###テンプレ4: service 実装
- 目的: 指定の集計 or 自動判定ロジックを実装する
- 前提: idempotent で再実行可能
- 出力: service function, tests, edge cases, transaction 方針
